# 向量数据库

## 1 向量数据库解决什么问题

在 RAG 里，向量数据库主要负责：

```text
文档 / 图片 / 商品 / 对话内容
        ↓
Embedding 模型转成向量
        ↓
写入向量数据库
        ↓
用户问题也转向量
        ↓
相似度检索 TopK
        ↓
把检索结果放进 LLM 上下文
        ↓
生成回答
```
它解决的是：**如何在大量非结构化数据中，快速找到语义相似内容**。


- 比如用户问：平台部署 CN/US 多环境怎么设计？
- 传统关键词搜索可能只能搜到包含 CN、US、环境 的文档。
- 向量检索可以找到语义相关的内容，比如：
    ```text
        “region + environment 配置模型”
        “多区域多环境路由设计”
        “cn-prod / us-staging 管理平台架构”
    ```

## 2 业界常见向量数据库分类

### 2.1 专用型向量数据库
专门为向量检索设计的，适合正式 RAG、AI 搜索、推荐系统。

| 产品                  | 类型        | 主要特点                     | 适合场景                    |
| ------------------- | --------- | ------------------------ | ----------------------- |
| **Milvus / Zilliz** | 开源 + 云服务  | 高性能、适合大规模向量检索            | 企业级 RAG、大数据量、多模态检索      |
| **Pinecone**        | 托管云服务     | 运维简单、Serverless、生产友好     | 海外 SaaS、快速上线、少运维团队      |
| **Qdrant**          | 开源 + 云服务  | Rust 实现、过滤能力强、部署轻量       | 中小团队 RAG、私有化部署、实时检索     |
| **Weaviate**        | 开源 + 云服务  | 内置 schema、GraphQL、混合搜索友好 | 知识库、语义搜索、AI 应用平台        |
| **Chroma**          | 开源轻量级     | 上手简单，适合本地开发              | Demo、Notebook、个人知识库、PoC |
| **LanceDB**         | 开源 / 本地优先 | 面向 AI 数据湖、嵌入式、本地文件友好   | 本地 RAG、离线数据处理、多模态实验   |

- Milvus 官方定位是面向大规模非结构化数据相似性搜索的开源向量数据库；
- Qdrant 官方强调向量相似搜索、过滤、混合查询和高级检索；
- Pinecone 官方主打托管式、生产级、可扩展向量数据库服务。

### 2.2 传统数据库扩展型
这类不是“专门的向量数据库”，而是在已有数据库里增加向量能力。

| 产品                              | 类型           | 主要特点            | 适合场景                |
| ------------------------------- | ------------ | --------------- | ------------------- |
| **PostgreSQL + pgvector**  | 关系型数据库扩展   | 业务数据和向量放一起，架构简单 | 中小规模 RAG、业务系统内嵌知识检索 |
| **MySQL / MariaDB 向量能力**        | 关系型数据库增强     | 适合已有 MySQL 技术栈  | 简单语义搜索、轻量推荐         |
| **Redis Vector Search**         | 内存 / 缓存型     | 低延迟，适合实时场景      | 实时推荐、短期记忆、会话记忆      |
| **MongoDB Atlas Vector Search** | 文档数据库 + 向量搜索 | 文档数据和向量结合方便 | 文档型业务、内容搜索、应用数据检索  |

### 2.3 搜索引擎增强型

本质上是搜索引擎，但加入了向量检索能力，优势是关键词 + 向量混合搜索。

| 产品                | 类型        | 主要特点                   | 适合场景         |
| ----------------- | --------- | ---------------------- | ------------ |
| **Elasticsearch** | 搜索引擎      | BM25 关键词检索强，支持向量检索     | 企业搜索、日志、文档搜索 |
| **OpenSearch**    | 搜索引擎      | AWS 生态友好，支持 kNN / 向量搜索 | 云上搜索、混合检索    |
| **Vespa**         | 搜索 / 推荐引擎 | 检索、排序、推荐能力强            | 大规模搜索推荐系统    |
| **Solr**          | 搜索引擎      | 老牌搜索平台，也有向量能力          | 传统搜索系统升级     |

RAG 里很常见的最佳实践不是纯向量检索，而是：
```text
关键词检索 BM25 + 向量检索 + 重排序 Rerank
```
也就是
```text
用户问题
  ↓
关键词召回一批
  ↓
向量召回一批
  ↓
合并去重
  ↓
Reranker 重排
  ↓
Top N 给大模型
```

所以如果公司已有 Elasticsearch / OpenSearch，不一定要推倒重来，可以先做混合检索升级。

### 2.4 云厂商托管型
| 产品                                                                | 云生态             | 适合场景                       |
| ----------------------------------------------------------------- | --------------- | -------------------------- |
| **Azure AI Search Vector Search**                  | Microsoft Azure | 已在 Azure 上使用 OpenAI / 企业搜索 |
| **Amazon OpenSearch / Aurora pgvector / Bedrock Knowledge Bases** | AWS       | AWS 生态内做 RAG          |
| **Google Vertex AI Vector Search**                        | Google Cloud    | GCP 生态、推荐、语义搜索     |
| **阿里云向量检索 / AnalyticDB / Elasticsearch 向量能力**   | 阿里云           | 国内业务、阿里云生态          |
| **腾讯云 / 火山 / 百度相关向量检索产品**                | 国内云             | 国内合规、低延迟、国产云生      |

如果你的业务分为国内 CN + 海外 US，这类云厂商方案要重点考虑：
- 数据合规 ：国内数据是否必须留在中国
- 网络延迟：Embedding 服务、向量库、LLM 是否在同一区域
- 运维成本：是否愿意自建 Milvus / Qdrant
- 云绑定：是否接受和 Azure / AWS / 阿里云绑定
